2. tests/core/detect_union.test.ts — 3 failing tests, TWO DISTINCT real issues, neither is the
   mirror_x/mirror_y problem above:

   2a. Union rebuild after rotate_pages()/delete_pages() does not reapply the FULL_PAGE_FRAC
   exclusion (spec §8). `model.document.union` is correctly computed (fallback full-page boxes
   excluded) at detect time, but after `rotate_pages()` or `delete_pages()` the rebuilt union comes
   back as the raw bounding box of ALL boxes including full-page fallbacks — e.g. rotate case:
   expected `{x0:20,y0:20,x1:280,y1:250}`, got `{x0:0,y0:0,x1:260,y1:300}` (test lines 100-102);
   delete case: expected `{x0:10,y0:10,x1:110,y1:210}`, got `{x0:0,y0:0,x1:198,y1:295}` (lines
   126-128). This is a genuine correctness bug (data quality of the auto-crop union after any
   rotate/delete on a document with a detected union), not a stub/WIP issue. Root cause not yet
   diagnosed — likely wherever rotate_pages()/delete_pages() rebuild `document.union` in
   `src/core/model.ts`, that rebuild path skips the same FULL_PAGE_FRAC filter the initial
   `_compute_detection_union` (model.ts ~line 366) applies.
   Severity: HIGH (silent data-quality regression, not a crash). Owner: unclear — closest existing
   task is T5 (auto-detect), but T5's brief (00_README_MASTER.txt) is scoped to progress-freeze/jump
   and outlier tolerance, not this. May need its own bug ID.

   2b. NORMAL-mode detection uses a PDF text-layer fast path that the test explicitly says must
   never be used. `src/core/model.ts:348-350` (`_detect_each_page`): "NORMAL pages: derive the
   content box from the PDF text layer ... Falls back to the Sauvola image path only when there is
   no usable text layer" — deliberately mirrors desktop `detect.py normal_page_box`, implemented via
   `PdfRendererAdapter.detect_text_box()` in `src/pdf/loader.ts:324`. The test file's own header
   comment (detect_union.test.ts:1-3) says the opposite is required: "NORMAL-mode detection must use
   the ink path (detect_content_box), never a pdf.js text-layer box — advance-based text boxes
   inflate x1 (right-margin regression)" and its first test asserts `text_calls.n` stays 0 (it's 2).
   Neither side is obviously wrong without more context — desktop parity (text-layer fast path,
   matches Python `detect.py`) vs. a specific "right-margin regression" bug report that this test
   was apparently written to codify as a fix (TDD-style) but whose corresponding model.ts change was
   never made. The test's own comment cites "spec-web §W2 row 10" for this, which is WRONG even
   under the doc's history — row 10 has only ever been "Same-size split windows" (added same commit,
   9cf743d) or didn't exist (pre-9cf743d table had 9 rows total, W2 row 10 was never
   detection-related). This citation was probably a placeholder guess made before the final row
   numbering landed, and neither the fix nor its doc row were ever completed. No existing spec-web
   §W2 row documents NORMAL-mode text-layer-vs-ink-path detection at all — this behavior (whichever
   way it's decided) is currently UNDOCUMENTED.
   Severity: HIGH if the "right-margin regression" bug is real on production PDFs (auto-detect boxes
   would be wrong on native/NORMAL documents) — needs someone to actually reproduce it against a
   real PDF with justified/wide text runs before deciding which side is correct. Do not just delete
   the failing test to go green; the comment reads like a genuine bug report, not noise.
   Owner: unclear, likely T5 (auto-detect) or a new bug ID; needs a spec-web §W2 row once resolved
   either way.
