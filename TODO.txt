    OPEN originally flagged 2026-07-05 — SCANNED-mode perf: navigation,
    Auto-detect and filters are slow vs desktop. Diagnosis (verified, not from any recent change —
    the pipeline is byte-identical to the pre-session HEAD): (a) every scanned work raster =
    pdf.js render at SRC_DPI 200 + OpenCV.js on the MAIN thread (§7a, deliberate); the vendored
    @techstark/opencv-js 4.10 build is single-threaded SCALAR WASM — zero SIMD in the binary
    (checked) — ≈2–5× slower than desktop's native OpenCV for the same Sauvola/despeckle/bilateral;
    threads impossible (no SharedArrayBuffer on Pages). (b) filters are lazy-on-view (16c1b6d):
    desktop hides processing in its §14 background tick loop, web pays it per cache-miss at
    NAVIGATION time, LRU=16, prefetch ±1. (c) dewarp falls back to 1-thread WASM if WebGPU init
    fails — unverified on a real GPU. Ranked levers, none violating the no-worker/no-threads
    rules: (1) idle background pipeline warming the work cache for the Pages selection (port of
    desktop §14 root.after pattern; biggest structural win), (2) WASM-SIMD opencv.js build swap
    (~2–4× CV math; dependency change, verify numerics on a real scan), (3) one-time console log
    of the ONNX EP actually engaged, (4) prefetch depth ±2–3. Do NOT resize/alter detection
    boxes math — algorithm matches desktop (user rule).
